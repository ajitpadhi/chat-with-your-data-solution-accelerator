### BUG-0061 — durable Event Grid MI fix

- **Root cause (confirmed):** RBAC replication latency, not creation ordering. `dependsOn` orders resource *creation*, but Azure RBAC is eventually consistent, and the **system-assigned** topic MI does not exist until the AVM `system-topic` module completes — so the `Storage Queue Data Message Sender` grant was authored moments before Event Grid's **synchronous** delivery-authorization preflight, with ~0 s replication lead → 401.
- **MACAE comparison:** MACAE uses no Event Grid at all; it handles grant-then-use via idempotent azd `postprovision` `az` scripts (post-ARM, replication settled) + a documented replication-delay retry. No pattern to copy directly.
- **Fix (durable, mirrors the existing-topic reuse path):** deliver as the **shared UAMI** (`id-<SUFFIX>`, exists at t=0) instead of the topic's system MI. Three coordinated edits in [../infra/main.bicep](../infra/main.bicep): (a) AVM `system-topic` `managedIdentities.userAssignedResourceIds: [ userAssignedIdentity.outputs.resourceId ]`; (b) `eventGridQueueSenderRole` grants the role to the UAMI on the storage account with **no** dependency on the topic, so ARM creates it at the start of the deployment and it replicates through the multi-minute provisioning tail before the preflight runs; (c) `blobCreatedSubscription.deliveryWithResourceIdentity.identity` → `UserAssigned`. `az bicep build` exit 0 (only pre-existing BCP081/BCP334 warnings elsewhere).
- **Rejected alternatives:** `deploymentScripts` grant-then-wait (needs `allowSharedKeyAccess=true` backing storage — conflicts with ADR 0002); moving the subscription into a `postprovision` hook (kept as a backstop, unneeded once the grant is UAMI-based and topic-independent).

**Two pre-existing local-dev-compose defects discovered** (both invisible to the working cloud deploy, both unrelated to the root reorg — the reorg preserved the same relative mount semantics):

- **Functions container — `ModuleNotFoundError: No module named 'functions'`.** `function_app.py` imports `from functions.<sub>.blueprint import …` (nested-package style). `Dockerfile.functions` builds the correct deploy layout (`/home/site/wwwroot/functions/<sub>/` + a generated `functions/__init__.py`, `PYTHONPATH=/home/site/wwwroot`) — which is why the **cloud** image runs and the cloud `/api/health` passed. But `docker-compose.dev.yml` then volume-mounts `../src/functions:/home/site/wwwroot`, which **flattens** the package (blueprints land directly at the deploy root, no `functions/` dir) and shadows the generated `__init__.py`. The host indexes zero HTTP routes (`No HTTP routes mapped`). The Dockerfile header even *claims* the compose mount "reuses the installed dependencies while serving live source" — but the mount as written cannot reproduce the nested layout with a single bind. Fix direction: mount the blueprint subpackages **under** `/home/site/wwwroot/functions/<sub>` (and `function_app.py` + `host.json` at the root), preserving the image's baked `functions/__init__.py`.
- **Backend container — cannot auth to local `pgvector`.** `PostgresClient._password_provider` always fetches an **Entra ID token** (`get_token(AadScope.POSTGRES_FLEX)`) and passes `user=postgres_admin_principal_name` — the provider is **AAD-only by design** (managed identity, no passwords / no Key Vault, per the standing hard rules). The local `pgvector/pgvector:pg16` container only speaks **password** auth and cannot validate an Entra token, and there is no `az`/managed identity inside the backend container (`CredentialUnavailableError: Azure CLI not found on path`). So the local full-stack backend↔DB path is **not functional with the current AAD-only provider** — a genuine limitation, not a quick config fix.
